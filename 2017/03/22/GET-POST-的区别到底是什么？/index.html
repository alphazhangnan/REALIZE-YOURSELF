GET POST 的区别到底是什么？

发表于 2017-03-22 | 阅读次数

通常情况下，GET 用来请求数据，POST 用来向服务器提交数据，但区别仅限于此？面试官想听的肯定不是这个模棱两可的答案。那就详细总结一番。

最普遍的答案

既然是最普遍的答案，受用性也是极高，出现频率奇高，但真实性的没有人去考量。
1.GET 使用 URL 或者 Cookie 传参， POST 将数据放在 body 中
2.GET 的 URL 会有长度上的限制，POST 的数据没有长度的限制
3.POST 比 GET 安全，因为在地址栏上不可见提交信息
以上只是最浅显的区别，甚至还存在了一些错误！下面一步步来纠错。

GET 和 POST 与数据如何递没有关系

在思考这个问题之前，先回归到《鸡和蛋》的概念中，GET 和 POST实质上是一样的。
1.首先 GET 和 POST 是HTTP协议定义的方法，HTTP是建立在TCP/IP(传输层)基础上的，使用什么样的方法与应用层的数据如何传输没有相互的关系。
2.HTTP 协议并没要求，POST必须放在 body 内。GET和POST能做的事情是一样一样的。你要给GET加上request body，给POST带上url参数，技术上是完全行的通的。但，能不能正确返回是个未知数。
3.现在Web Server都是支持 GET 和 POST 包含 body 这样的请求的，虽然浏览器是无法发出这样的请求，但是服务器并不只是只有浏览器能使用。

那区别在哪里？

1.回归TCP层面，毕竟这是基础。最重要的两个区别的：
GET 请求时产生一个 TCP 数据包，浏览器发起 GET 请求后，服务器直接回返回200之类的；
2.POST 请求回产生两个 TCP 数据包，浏览器发起 POST 请求后，先发送报头信息，服务器返回100状态码后，再发送主体信息
3.HTTP也要遵循语义化规则，不同的方法应该做符合语义化的事情
4.在网络良好的情况下，GET 和 POST 的时间差基本可以忽略不计
5.当然也并非所有的浏览器POST会发送两次数据包，Firefox就只发一次

HTTP 协议对 GET 和 POST 都没有长度限制

在HTTP协议标准中：报头和 body 都没有长度的要求。而对于URL长度上的限制，其实更多的应该是浏览器和服务器减轻负担所使用的策略，两者商定了一种”对策”！

POST 未必比 GET 安全，还是要看服务器

前面也说过了，GET/POST 的参数，没什么大的区别，所以对于安全性考虑：
1.URL参数虽然经过 Encode/Base64 编码，变得不可读，但是了解一下解码(Decode/a to b)，反解参数数据也不难。同样的例子，就像你破解隔壁小姐姐的Wifi密码一样。
2.往高级一点说，截包重发。

总结

#GET/POST，本质上的无区别，实则是在提醒程序员们应该注重网络安全。#
